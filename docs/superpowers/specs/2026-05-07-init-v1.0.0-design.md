# Initialize project as v1.0.0

**Date:** 2026-05-07
**Status:** Fulfilled by Cluster 10 (2026-05-11)

## Goal

Re-baseline the entire `@storely` monorepo at version `1.0.0`. Every workspace
package and the root manifest currently reads `6.0.0-alpha.4`; after this change
all 23 `package.json` files must read `1.0.0`.

## Context

- Monorepo managed with pnpm (`pnpm-workspace.yaml`).
- 22 workspace packages across `core/`, `serialization/`, `compression/`,
  `encryption/`, `storage/`, plus `website/`.
- Inter-package dependencies use the `workspace:` protocol, so internal `deps`
  /`peerDeps` strings do not encode version numbers and need no edits.
- `scripts/version-sync.ts` already exists: it reads the root `package.json`
  version and writes it into every workspace package's `package.json`. The root
  `package.json` exposes it as `pnpm version:sync`.
- Nothing has ever been published to npm, so re-using `1.0.0` cannot collide
  with prior releases.
- No `.git` directory is present; this change does not involve commits or tags.

## Approach

Use the existing `version-sync` script — it is the project's documented path
for keeping all packages in lockstep, and going through it preserves any
formatting conventions baked into the script.

## Steps

1. **Edit root `package.json`:** change `"version": "6.0.0-alpha.4"` to
   `"version": "1.0.0"`. No other field changes.
2. **Run `pnpm version:sync`:** propagates `1.0.0` to all 22 workspace
   `package.json` files. Expected output: 22 updated, 0 skipped.
3. **Run `pnpm install`:** refreshes `pnpm-lock.yaml` so lockfile entries for
   the workspace packages reflect `1.0.0`.
4. **Verify:** list every package's `name | version` and confirm all 23 read
   `1.0.0` exactly.

## Out of scope

- No `git init`, commit, or tag. (No git repo exists; the user has not asked
  for one.)
- No CHANGELOG entry, release notes, or publish.
- No edits to historical documents under `docs/superpowers/specs/` or
  `docs/superpowers/plans/` that mention `6.0.0-alpha.4`. Those are
  point-in-time records and should stay accurate to when they were written.
- No changes to `.github/workflows/release.yml`. Its only `6.0.0-alpha.4`
  reference is an example string in a `workflow_dispatch` input description,
  not a hardcoded version.

## Risks and mitigations

- **Lockfile drift.** If `pnpm install` is skipped, `pnpm-lock.yaml` will
  reference the old version for workspace packages. Mitigation: step 3 above.
- **Unintentional reformatting of `package.json` files.** The `version-sync`
  script preserves indentation per file (it detects `\t` vs spaces), so this
  should not be a concern, but the verification step will surface any oddities
  if a `package.json` ends up rewritten unexpectedly.

## Success criteria

- `cat package.json | jq -r .version` prints `1.0.0`.
- For every workspace package, `jq -r .version package.json` prints `1.0.0`.
- `pnpm-lock.yaml` references `1.0.0` for the workspace packages.
- No other files have been modified.
